## 一、模块现状分析（AS-IS）

### 1.1 现状总览

当前面试模块存在严重的"脚手架半成品"状态——**数据库表已定义但后端逻辑全部走 mock，管理端完全缺失，门户端只有基础页面**。具体问题如下：

| 层面 | 现状描述 | 问题程度 |
| --- | --- | --- |
| 数据库 | 已定义 9 张表，核心表完整，但部分辅助表缺少字段 | 中 |
| 后端 Controller | `PortalInterviewController` 仅 2 个接口，全部硬编码 Map 返回 | 严重 |
| 后端 Entity/Mapper | 仅 4 个实体类（Question/Category/Experience/ResumeTemplate），其他 5 张表无 Entity | 严重 |
| 后端 Service | **完全缺失** | 严重 |
| 后端 VO/DTO | **完全缺失** | 严重 |
| 管理端 CMS | **完全缺失**（无 Controller、无管理页面、无 API） | 严重 |
| 门户前端 | 仅有 1 个 `InterviewPage.vue` 首页，无题目详情、练习、面经等页面 | 严重 |
| 门户前端 API | `interview.ts` 仅 3 个方法（2 个未实现） | 严重 |
| 字段对齐 | Entity 字段与数据库表字段**基本对齐**，但前端类型定义和后端 VO 完全错位 | 中 |

## 二、重构目标与设计原则（TO-BE）

### 2.1 业务目标

1. **闭环的面试练习**：从题目浏览 → 练习/代码提交 → 查看题解 → 收藏回顾的全链路
2. **丰富的面经社区**：用户可发布真实面试经验，支持评论、点赞
3. **高质量简历模板**：按岗位/行业分类的可下载简历模板资源
4. **完整的管理后台**：题目/分类/面经/模板的 CMS 全量管理

### 2.2 技术设计原则

| 原则 | 说明 |
| --- | --- |
| **复用现有架构** | 沿用 Spring Boot 3.3 + MyBatis-Plus + Spring Security 6 + JWT；前端沿用 Vue 3 + Element Plus/Vite |
| **三层架构对齐** | Entity（数据库）→ Mapper.xml（SQL）→ Service（业务）→ VO（前端返回）；每个实体有完整 CRUD |
| **前后端字段一致** | 后端字段使用驼峰命名（如 `acceptanceRate`），数据库下划线命名（`acceptance_rate`），通过 MyBatis-Plus `map-underscore-to-camel-case` 自动映射；前端 TS 类型与后端 VO 字段严格一致 |
| **分页统一** | 所有列表接口使用 `PageDomain` 继承查询参数，统一采用 `pageNum/pageSize` + `orderByColumn/isAsc` |
| **权限分层** | 匿名用户可读；登录用户可练习、点赞、收藏、发布面经；管理员可进入 CMS 审核 |
| **事务安全** | 涉及多表写入的操作（如删除面经同时删除点赞、评论）必须用 `@Transactional` |
| **字段对齐零冗余** | Entity 字段与数据库列一一对应；VO 按需裁剪；避免在 Entity 与 VO 之间字段名漂移 |

### 2.3 命名规范（与项目既有代码对齐）

| 元素 | 命名示例 | 备注 |
| --- | --- | --- |
| Controller | `PortalInterviewQuestionController` / `CmsInterviewQuestionController` | 门户 Controller 在 `com.moyun.portal.controller`；CMS Controller 在 `com.moyun.ext.cms.controller` |
| Service | `IPortalInterviewQuestionService` / `PortalInterviewQuestionServiceImpl` | 接口 + 实现分离 |
| Mapper | `PortalInterviewQuestionMapper` | 放在 `com.moyun.portal.mapper` |
| VO | `InterviewQuestionVO` / `InterviewCategoryVO` | 放在 `com.moyun.portal.domain.vo` |
| Query | `InterviewQuestionQuery` | 放在 `com.moyun.portal.domain.query` |
| Entity | `PortalInterviewQuestion` | 放在 `com.moyun.portal.domain.entity` |
| 表前缀 | `portal_interview_*` | 保持既有约定 |
| CMS API 路径 | `/cms/interview/question/list` | `/cms/interview/**` 统一前缀 |
| 门户 API 路径 | `/portal/interview/question/{id}` | `/portal/interview/**` 统一前缀 |

#### （1）题目分类表 `portal_interview_category`（既有，保持不变）

```sql
CREATE TABLE `portal_interview_category` (
    `id` BIGINT NOT NULL AUTO_INCREMENT COMMENT '主键',
    `name` VARCHAR(200) NOT NULL COMMENT '分类名称',
    `slug` VARCHAR(200) DEFAULT NULL COMMENT 'URL标识（SEO友好）',
    `description` TEXT COMMENT '分类描述',
    `icon` VARCHAR(500) DEFAULT NULL COMMENT '图标URL',
    `sort` INT DEFAULT 0 COMMENT '排序',
    `question_count` INT DEFAULT 0 COMMENT '题目数量（冗余字段，定时更新）',
    `status` VARCHAR(20) DEFAULT 'active' COMMENT '状态:active,inactive',
    `create_by` VARCHAR(64) DEFAULT '' COMMENT '创建者',
    `create_time` DATETIME DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
    `update_by` VARCHAR(64) DEFAULT '' COMMENT '更新者',
    `update_time` DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间',
    `remark` VARCHAR(500) DEFAULT NULL COMMENT '备注',
    PRIMARY KEY (`id`),
    KEY `idx_status` (`status`),
    KEY `idx_slug` (`slug`)
```

### 4.5 Service 设计要点

1. **PortalInterviewQuestionServiceImpl**
   - `selectQuestionPage`: 分页 + 分类/难度/关键词筛选 + 排序
   - `selectQuestionById`: 查详情 + 关联公司列表 + 当前用户点赞/收藏/做题状态
   - `submitAnswer`: 插入 submission + 更新 attempt 表 + 异步更新题目 acceptance_rate
   - `toggleLike`: 幂等切换点赞（插入/删除 question_like）+ 更新 question.like_count
   - `toggleBookmark`: 同上

2. **PortalInterviewExperienceServiceImpl**
   - `selectExperiencePage`: 分页 + 公司/年份筛选 + 置顶优先排序
   - `selectExperienceById`: 查详情 + view_count 原子自增
   - `publishExperience`: 插入面经（status = pending）
   - `toggleLike`: 切换点赞

3. **PortalInterviewHomeController**
   - `getHome`: 聚合调用 5 个 Service 的 `listTopN` 方法，组合成 HomeDataVO 返回

4. **计数异步更新策略**
   - `acceptanceRate`: 每次新提交后用异步任务重新计算该题 `is_success=1` 的比例
   - `category.questionCount`: 使用定时任务或数据库触发器维护（简单起见，查询时用子查询）
   - `viewCount`: 用 `UPDATE table SET view_count = view_count + 1 WHERE id = ?` 保证原子性

## 十二、分页设计规范（v2.1.0 新增）

本节定义面试空间所有 Controller 列表接口统一的分页实现方案，确保字段命名、组件调用与项目既有风格（`BaseController` / `PageDomain` / `PageUtils` / `TableSupport`）完全一致。

### 12.1 三层架构关系

```
+-----------------+        +-----------------+        +-----------------+
|  Controller     |  --->  |   PageUtils     |  --->  |   TableDataInfo |
|  (BaseController|        |  (startPage())  |        |  (rows + total) |
|   子类)         |        |                 |        |                 |
+-----------------+        +-----------------+        +-----------------+
        |                         |
        v                         v
+-----------------+        +-----------------+
| Query extends   |        |  TableSupport   |
| PageDomain      |        | (解析请求参数)  |
+-----------------+        +-----------------+
```

| 组件 | 位置 | 职责 |
| --- | --- | --- |
| `BaseController` | `com.moyun.core.base.BaseController` | 提供 `startPage()` / `getDataTable()` 模板方法，供所有列表 Controller 直接继承调用 |
| `PageDomain` | `com.moyun.core.base.page.PageDomain` | 封装分页/排序字段（`pageNum` / `pageSize` / `orderByColumn` / `isAsc`）；所有 `*Query` 对象继承它 |
| `TableSupport` | `com.moyun.core.base.page.TableSupport` | 解析 HTTP 请求参数，构造 `PageDomain` 并调用 MyBatis-Plus 分页 |
| `PageUtils` | `com.moyun.util.bean.PageUtils` | 通用分页工具；`startPage()` 内部调用 `TableSupport.buildPageRequest()` |
| `TableDataInfo` | `com.moyun.core.base.TableDataInfo` | 列表返回体，包含 `code / msg / rows / total` 四个字段 |

### 12.2 字段命名规范（重要！）

以下字段名在 **后端 Query / VO** 与 **前端 TS Query** 中必须完全一致（驼峰），禁止混用 `page` / `pageNo` 等其他命名：

| 字段名 | 类型 | 默认值 | 说明 |
| --- | --- | --- | --- |
| `pageNum` | Integer | `1` | 当前页码（不要使用 `page` / `pageNo`） |
| `pageSize` | Integer | `10` | 每页条数（不要使用 `page_size` / `limit`） |
| `orderByColumn` | String | `""` | 排序字段（表列的驼峰，如 `createTime`） |
| `isAsc` | String | `"asc"` | 排序方向（`"asc"` 升序 / `"desc"` 降序） |

> 兼容性说明：前端旧代码仍可能传入 `page` / `pageNo`，Controller 读取时应优先使用 `PageDomain` 标准字段；Service 层通过 `TableSupport.buildPageRequest()` 解析，可兼容上述变体（在工具层做归一化）。

### 12.3 Controller 调用范式

**推荐写法**（继承 `BaseController` + 统一分页）：

```java
@RestController
@RequestMapping("/portal/interview/question")
public class PortalInterviewQuestionController extends BaseController {

    @Autowired
    private IPortalInterviewQuestionService questionService;

    @GetMapping("/list")
    @Anonymous
    public TableDataInfo list(InterviewQuestionQuery query) {
        // 1. 开启分页：内部自动读取 request 中的 pageNum/pageSize/orderByColumn/isAsc
        startPage();
        // 2. 业务查询（返回 List<VO>）
        List<InterviewQuestionVO> list = questionService.selectQuestionList(query);
        // 3. 组装 TableDataInfo（包含 rows + total）
        return getDataTable(list);
    }
}
```

### 12.7 反模式（禁止使用）

| ❌ 错误写法 | 原因 | ✅ 正确写法 |
| --- | --- | --- |
| 手写 `SELECT * FROM ... LIMIT ?, ?` | 绕过 MyBatis-Plus，分页实现不统一 | 使用 `startPage()` + `selectList()` |
| 返回 `{ page, list }` 或 `{ items, total }` | 前端无法复用通用分页组件 | 返回 `TableDataInfo`（rows + total） |
| 在 Controller 内部私写 `class PageUtils { ... }` | 与全局 `PageUtils` 冲突，后续升级易出错 | 删除私有类，import `com.moyun.util.bean.PageUtils` |
| 参数名为 `page / pageNo / size / limit` | 与 `PageDomain` 字段不匹配 | 使用 `pageNum / pageSize` |

## 十三、通用标签系统设计（v2.1.0 新增）

本节定义面试空间中**标签（Tag）**与**多实体绑定（Entity-Tag）**的设计，支持「标签→题目/面经/简历模板等多实体关联」以及「热门标签排行」查询。设计目标：**与项目既有的 `portal_tag` 表对齐、最小侵入、便于扩展到其他模块**。

### 13.1 数据库表设计

#### （A）`portal_tag` 表（扩展）

| 新增字段 | 类型 | 默认值 | 说明 |
| --- | --- | --- | --- |
| `module` | VARCHAR(32) | `NULL` | 所属模块：`common`（通用）/ `article` / `book` / `interview_question` / `interview_experience` 等；为 `NULL` 视为 `common` |
| `reference_count` | BIGINT UNSIGNED | `0` | 被引用次数（= `portal_entity_tag` 中该 tag_id 的行数，用于"热门标签"排行） |

对应 DDL（在 `portal_module_ddl_v2.sql` 中已有）：

```sql
ALTER TABLE `portal_tag`
    ADD COLUMN `module`          VARCHAR(32)     DEFAULT NULL COMMENT '所属模块' AFTER `status`,
    ADD COLUMN `reference_count` BIGINT UNSIGNED DEFAULT 0    COMMENT '被引用次数' AFTER `module`,
    ADD INDEX `idx_module` (`module`),
    ADD INDEX `idx_reference_count` (`reference_count` DESC);
```

#### （B）`portal_entity_tag` 表（新增）

```sql
CREATE TABLE `portal_entity_tag` (
    `id`          BIGINT UNSIGNED NOT NULL AUTO_INCREMENT COMMENT '主键',
    `tag_id`      BIGINT UNSIGNED NOT NULL COMMENT '标签ID（引用 portal_tag.id）',
    `entity_type` VARCHAR(32)     NOT NULL COMMENT '实体类型',
    `entity_id`   BIGINT UNSIGNED NOT NULL COMMENT '实体ID',
    `sort`        INT             DEFAULT 0  COMMENT '排序',
    `create_time` DATETIME        DEFAULT NULL COMMENT '创建时间',
    PRIMARY KEY (`id`),
    UNIQUE KEY `uk_tag_entity`      (`tag_id`, `entity_type`, `entity_id`),
    KEY        `idx_entity`         (`entity_type`, `entity_id`),
    KEY        `idx_entity_create`  (`entity_type`, `create_time`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='通用实体标签关联表';
```

**关键字段解释**：

| 字段 | 作用 |
| --- | --- |
| `entity_type` | 标识被打标签的实体，预定义值：`interview_question` / `interview_experience` / `interview_resume_template` / `article` / `book` 等 |
| `uk_tag_entity` 唯一索引 | 保证同一实体在同一标签下仅关联一次（防重复绑定） |
| `idx_entity` 索引 | 用于 `SELECT * FROM portal_entity_tag WHERE entity_type = ? AND entity_id = ?` 的"查某实体所有标签"场景 |
| `idx_entity_create` | 分页按时间排序查询 |

### 13.3 三个核心 API

#### ① 绑定标签（POST）

```
POST /portal/tag/bind

请求体：
{
  "tagId": 100,
  "entityType": "interview_question",
  "entityId": 42,
  "sort": 0
}

返回：
{
  "code": 200,
  "msg": "绑定成功",
  "data": { "bound": true, "newReferenceCount": 15 }
}
```

**幂等处理**：若 `(tag_id, entity_type, entity_id)` 已存在，不插入新行，直接返回成功；返回时同步更新 `portal_tag.reference_count`（= 当前该 tag 在 entity_tag 中的行数）。

#### ② 解绑标签（DELETE）

```
DELETE /portal/tag/unbind?tagId=100&entityType=interview_question&entityId=42

返回：
{ "code": 200, "msg": "解绑成功", "data": { "bound": false } }
```

#### ③ 按实体查所有标签

```
GET /portal/tag/list/{entityType}/{entityId}

返回：
{
  "code": 200,
  "rows": [
    { "id": 100, "name": "高频题", "slug": "high-frequency", "sort": 1 },
    { "id": 201, "name": "Java", "slug": "java", "sort": 2 }
  ]
}
```

SQL（`PortalEntityTagMapper.xml`）：

```sql
SELECT t.id, t.name, t.slug, t.status, et.sort
  FROM portal_entity_tag et
  JOIN portal_tag t ON t.id = et.tag_id
 WHERE et.entity_type = #{entityType}
   AND et.entity_id   = #{entityId}
 ORDER BY et.sort ASC, et.create_time ASC;
```

### 13.4 热门标签排行（首页 / 广场）

```
GET /portal/tag/hot?module=interview_question&limit=10

返回：
{
  "code": 200,
  "rows": [
    { "id": 100, "name": "高频题", "slug": "high-frequency", "referenceCount": 1523 },
    { "id": 201, "name": "Java",    "slug": "java",          "referenceCount":  821 }
  ]
}
```

SQL 两种实现：

| 实现 | 优点 | 缺点 |
| --- | --- | --- |
| **直接查 `portal_tag.reference_count`**（推荐） | 性能最高（走索引 `idx_module + idx_reference_count`），适合读多写少 | 需要在绑定/解绑时同步更新计数 |
| **COUNT 子查询**（简单实现） | 完全准确、无需维护计数 | 数据量大时性能差 |

推荐实现（直接查计数列）：

```sql
SELECT id, name, slug, reference_count
  FROM portal_tag
 WHERE status = '0'
   AND (module = #{module} OR module IS NULL)
 ORDER BY reference_count DESC, sort ASC, id ASC
 LIMIT #{limit};
```

### 13.5 reference_count 更新策略

`reference_count` 是冗余计数列，更新有两个方案：**实时同步**（简单，推荐）和 **定时任务**（兜底）。

**实时同步（Service 层）**：

```java
@Service
public class PortalTagServiceImpl implements IPortalTagService {

    @Transactional(rollbackFor = Exception.class)
    public void bind(Long tagId, String entityType, Long entityId, Integer sort) {
        // 1. 先查是否已绑定（唯一索引兜底）
        boolean exists = portalEntityTagMapper.exists(tagId, entityType, entityId);
        if (exists) {
            return; // 幂等
        }
        // 2. 插入关联记录
        portalEntityTagMapper.insert(tagId, entityType, entityId, sort);
        // 3. 原子更新 reference_count（COUNT(*) 子查询）
        portalTagMapper.updateReferenceCount(tagId);
    }

    @Transactional(rollbackFor = Exception.class)
    public void unbind(Long tagId, String entityType, Long entityId) {
        int affected = portalEntityTagMapper.deleteOne(tagId, entityType, entityId);
        if (affected > 0) {
            portalTagMapper.updateReferenceCount(tagId);
        }
    }
}
```

Mapper XML：

```xml
<!-- 原子更新 reference_count = 当前该 tag 在 entity_tag 中的行数 -->
<update id="updateReferenceCount">
  UPDATE portal_tag
     SET reference_count = (SELECT COUNT(*) FROM portal_entity_tag WHERE tag_id = #{tagId})
   WHERE id = #{tagId}
</update>
```

### 13.6 与面试模块集成的三种场景

| 场景 | 使用方式 |
| --- | --- |
| **题目页** | 每道题目有 tags 字符串（兼容旧数据）；同时绑定 `portal_entity_tag` 做准确关联；展示时以 entity_tag 为主、tags 字符串为辅 |
| **面经页** | 面经发布时可选择 / 输入多个标签；标签写入 `portal_tag(module='interview_experience')`，同时写入关联表 |
| **面试首页** | 在 HomeDataVO 中追加 `hotTags: InterviewTagVO[]`；服务调用 `selectHotTags('interview_question', 12)` 做热门排行 |

### 13.7 扩展到其他模块

由于 `entity_type` 字段做了字符串枚举化，后续其他模块（文章、书籍）使用通用标签系统时，**无需新建表**：

1. 在 `EntityType` 类中追加常量 `ARTICLE = "article"`
2. 在 `portal_tag.module` 中约定 `module = 'article'`
3. 调用同一套 `bind / unbind / list` API 即可
