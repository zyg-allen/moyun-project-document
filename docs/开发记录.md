#### 1. 前后台独立认证体系

**需求**: 前台用户与后台管理员使用独立的用户表和认证体系

**实现方案**:
- 双 `SecurityFilterChain` 配置：
  - `SecurityConfig` - 后台管理认证（路径 `/system/**`、`/login` 等）
  - `PortalSecurityConfig` - 前台认证（路径 `/portal/**`）
- 独立用户表：
  - `sys_user` - 后台管理员表
  - `portal_user` - 前台用户表
- 独立认证服务：
  - `UserDetailsServiceImpl` - 后台用户认证
  - `PortalUserDetailsServiceImpl` - 前台用户认证
- 独立 Token 服务：
  - `TokenService` - 后台 Token 管理
  - `PortalTokenService` - 前台 Token 管理

### 第二阶段开发 - CMS内容管理系统（2026-05-27）

#### 1. CMS模块独立架构设计

**需求**: 创建独立的CMS模块，方便后续扩展其他应用（如算命、文件转换器等）

**设计方案**:
```
ext/
└── cms/
    ├── controller/       # CMS 管理控制器
    ├── domain/
    │   ├── dto/         # 数据传输对象
    │   ├── vo/          # 视图对象
    │   └── query/       # 查询对象
    ├── mapper/          # 数据访问层（复用portal模块）
    └── service/         # 业务逻辑层
        └── impl/
```

**架构优势**:
- 模块完全独立，便于维护和扩展
- 复用portal模块的实体类和mapper，避免重复代码
- 遵循若依框架代码风格
- 为后续添加新应用预留扩展路径

**相关文件**:

#### 14. 前端门户分页改造（2026-06-04）

**问题**:
- 前端门户分页功能不生效，MyBatis-Plus分页插件未正常工作
- 前后端分页响应格式不一致，导致前端无法正确解析分页数据
- 分页与导出功能无法区分，导出时数据不完整

**解决方案**:

1. **后端分页改造**:
   - 为所有需要分页的Mapper方法添加`Page<T>`参数作为第一个参数
   - 保留原有的XML SQL配置，确保MyBatis-Plus分页插件能自动拦截并添加LIMIT语句
   - 新增不分页的查询方法用于导出功能，通过方法重载区分分页/不分页场景
   - 统一返回MyBatis-Plus的`Page`对象，包含`records`（数据列表）、`total`（总数）、`current`（当前页）、`size`（每页大小）、`pages`（总页数）

2. **前端分页适配**:
   - 修改`client.ts`中的`httpGetList`函数，兼容处理多种分页响应格式
   - 新增支持`records`字段（MyBatis-Plus Page对象），优先使用
   - 保持原有对`rows`和`list`字段的支持，确保兼容性
   - 更新相关API调用，统一使用`httpGetList`处理分页数据

3. **涉及模块**:
   - 文章（Article）
   - 用户（User）
   - 分类（Category）
   - 标签（Tag）
   - 评论（Comment）
   - 收藏（Bookmark）
   - 关注（Follow）
   - 订单（Order）
   - 钱包（Wallet）
   - VIP（Vip）
   - 通知（Notification）
   - 友情链接（FriendLink）

**关键实现**:
```java
// Mapper层 - 分页查询
Page<PortalArticle> selectPortalArticlePage(Page<PortalArticle> page, @Param("params") ArticleQuery portalArticle);

// Mapper层 - 不分页查询（用于导出）
List<PortalArticle> selectPortalArticleList(@Param("params") ArticleQuery portalArticle);
```

```typescript
// client.ts - 兼容多种分页响应格式
const list = pageData.records || pageData.rows || pageData.list || [];
```

## 技术问题与解决方案

### 问题 1: Java 编译错误 - 找不到 update 方法

**错误信息**:
```
java: 找不到符号 方法 updatePortalFollow(com.moyun.portal.domain.PortalFollow)
```

**问题原因**: Service 接口、实现类和 Mapper 中缺少 update 方法

**解决方案**:
- 在 `IPortalFollowService` 接口添加 `updatePortalFollow` 方法
- 在 `PortalFollowServiceImpl` 中实现该方法
- 在 `PortalFollowMapper` 接口中添加对应的 mapper 方法
- 同样的问题也出现在 `PortalBookmark` 和 `PortalLike` 模块中，采用相同方式解决

### 问题 2: Git 分支分歧

**问题现象**: 本地分支与远程分支出现分歧

**解决方案**:
```bash
git fetch --all
git reset --hard origin/main
```

### 问题 3: 代码引用错误（旧包路径）

**问题原因**: 包结构优化后，代码中仍引用旧的包路径

**示例**:
- 旧路径: `com.moyun.common.core.redis.RedisCache`
- 新路径: `com.moyun.core.config.redis.RedisCache`

**解决方案**: 批量更新所有代码中的包路径引用

### 问题 4: 前台 API 路径与后台不匹配

**问题原因**: 前台 API 路径使用 `/api/` 或 `/` 前缀，后台控制器使用 `/portal/` 前缀

**解决方案**: 将前台所有 API 路径统一前缀为 `/portal/`

### 问题 5: 注册接口返回格式不一致

**问题**: 登录接口返回 `{ token, user }`，但注册接口只返回成功消息

**解决方案**: 修改 `PortalLoginController.register()` 方法，注册成功后自动登录，返回与登录接口一致的格式

### 问题 6: HomePage.vue 导入路径错误
**问题现象**: `getFriendLinks` 函数从错误的路径导入

**问题原因**: 从 `@/api/article` 导入，但该函数实际在 `@/api/friendLink` 中

**解决方案**: 修改导入路径为 `@/api/friendLink`

### 问题 7: 友情链接API响应格式不匹配

**问题现象**: 前端获取友情链接失败，期望返回 `response.data` 是 undefined

**问题原因**: 后端友情链接列表返回分页格式 `{code, msg, rows, total}`，但前端期望直接数组格式

**解决方案**:
- 修改 `friendLink.ts`，使用 `httpGetList` 替代 `httpGet`
- `HomePage.vue`，从 `response.data` 改为 `response.data.list`

### 问题 8: 前台门户分页不生效

**问题现象**:
- 分页查询返回全部数据，没有进行分页
- MyBatis-Plus分页插件未正常工作

**问题原因**:
- Mapper方法没有接收`Page<T>`参数，分页插件无法识别需要分页
- 分页响应格式不统一，前端无法正确解析

**解决方案**:
- 为所有分页查询的Mapper方法添加`Page<T>`作为第一个参数
- 统一使用MyBatis-Plus的`Page`对象返回分页数据
- 前端兼容处理`records`、`rows`、`list`多种字段格式

### 问题 9: 分页与导出功能冲突

**问题现象**:
- 导出功能使用分页查询，导致只导出当前页数据
- 数据不完整

**解决方案**:
- 新增不分页的查询方法用于导出
- 通过方法重载区分分页/不分页场景
- 分页使用`selectXxxPage`，导出使用`selectXxxList`

## 重要决策记录

### 决策 1: 前后台用户表独立 vs 公用

**背景**: 讨论前台用户和后台管理员是否使用同一张用户表

**方案对比**:

| 维度 | 独立表方案 | 公用表方案 |
|------|-----------|-----------|
| **数据隔离** | ✅ 完全隔离，安全性高 | ❌ 数据混合，需通过字段区分 |
| **认证体系** | ✅ 可独立配置认证策略 | ❌ 需在同一体系中处理不同用户类型 |
| **扩展性** | ✅ 前台用户可扩展更多字段 | ⚠️ 表结构会越来越复杂 |
| **维护成本** | ⚠️ 需维护两套用户体系 | ✅ 只需维护一套 |
| **业内实践** | ✅ 多数社区平台采用此方案 | 适用于小型系统 |

**最终决策**: **采用独立表方案**

**理由**:
- 数据安全隔离
- 便于后续扩展（如第三方登录、VIP体系等）
- 符合业内最佳实践

### 决策 2: Spring Security 双 FilterChain 配置

**背景**: 实现前后台独立认证

**方案**:
- 配置两个 `SecurityFilterChain` Bean
- 通过 `requestMatchers` 区分路径
- 各自使用独立的 `UserDetailsService` 和 `TokenService`

**配置要点**:
```java
// 后台安全配置
@Order(1)
@Bean
public SecurityFilterChain securityFilterChain(HttpSecurity http) {
    http.securityMatcher("/system/**", "/login", ...)
        // ... 后台认证配置
}

// 前台安全配置
@Order(2)
@Bean
public SecurityFilterChain portalSecurityFilterChain(HttpSecurity http) {
    http.securityMatcher("/portal/**")
        // ... 前台认证配置
}
```

### 决策 3: 第三方登录接入方案

**背景**: 规划接入微信登录、支付宝登录

**方案**: 策略模式 + 工厂模式

**设计**:
- `SocialLoginStrategy` 接口 - 定义第三方登录统一接口
- `WeChatLoginStrategy` - 微信登录实现
- `AlipayLoginStrategy` - 支付宝登录实现
- `SocialLoginFactory` - 工厂类，根据类型创建对应的策略实例

**优势**:
- 易于扩展新的第三方登录方式
- 各登录方式逻辑隔离
- 符合开闭原则
