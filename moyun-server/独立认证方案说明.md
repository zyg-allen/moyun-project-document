# 独立认证方案说明

## 方案概述

本项目采用**前后台独立认证**的方案，通过 `sys_user` 和 `portal_user` 两张表分别管理后台和前台用户，同时通过两套独立的 Spring Security 配置实现认证隔离。

## 核心实现

### 1. 双 SecurityFilterChain 配置

- **门户配置** (`PortalSecurityConfig`):
  - `@Order(1)`: 更高优先级，优先匹配
  - `securityMatcher("/portal/**")`: 只匹配门户路径
  - 使用 `portalUserDetailsServiceImpl` 加载门户用户
  - 使用 `portalAuthenticationManager` 处理门户认证

- **后台配置** (`SecurityConfig`):
  - `@Order(2)`: 较低优先级，后匹配
  - `securityMatcher` 排除门户路径
  - 使用原有的后台认证体系

### 2. 独立的 Token 管理

- **后台**: `TokenService` + `LOGIN_TOKEN_KEY`
- **门户**: `PortalTokenService` + `PORTAL_LOGIN_TOKEN_KEY`

两种 token 互不干扰，分别存储在 Redis 不同的 key 前缀下。

### 3. 用户认证流程

**后台用户登录:**
```
POST /login
    ↓
使用 sys_user 表验证
    ↓
生成后台 token
    ↓
存储到 login_tokens:
```

**门户用户登录:**
```
POST /portal/login
    ↓
使用 portal_user 表验证
    ↓
生成门户 token
    ↓
存储到 portal_login_tokens:
```

## API 接口

### 门户认证接口

| 接口 | 方法 | 说明 |
|------|------|------|
| `/portal/login` | POST | 门户用户登录 |
| `/portal/register` | POST | 门户用户注册 |
| `/portal/logout` | POST | 门户用户退出 |
| `/portal/user/info` | GET | 获取当前用户信息 |

### 公开访问的门户接口

- `GET /portal/articles` - 获取文章列表
- `GET /portal/articles/{id}` - 获取文章详情
- `GET /portal/categories` - 获取分类列表
- `GET /portal/tags` - 获取标签列表
- `GET /portal/comments` - 获取评论列表

其他门户接口需要认证。

## 用户关联

`portal_user` 表有 `user_id` 字段，可选关联到 `sys_user` 表，支持：

- 管理员用后台账号登录前台
- 操作日志记录审核人员信息
- 等等

## 优势

1. **数据隔离**: 后台和前台用户数据完全分离
2. **安全可靠**: 不会出现越权访问问题
3. **易于扩展**: 门户业务独立发展，不受后台影响
4. **维护简单**: 两套系统互不干扰
5. **符合规范**: 业内常见的 SaaS/社区系统架构

## 注意事项

1. **密钥配置**: 在 `application.yml` 中配置 `token.secret`
2. **Redis**: 需要确保 Redis 服务正常运行
3. **密码加密**: 都使用 BCrypt 加密存储
4. **角色体系**: 
   - 后台: 复杂的角色权限体系 (sys_role, sys_permission)
   - 门户: 简单的角色体系 (user/vip/admin)
